1) How do you check CPU & Memory usage?

In Linux, we can check CPU and memory usage using different commands. The most common commands are:

  • top – Shows real-time CPU and memory usage.
  • htop – Advanced version of top with better UI.
  • free -m – Shows memory usage in MB.
  • ps aux – Shows running processes.
  • vmstat – Shows system performance statistics.

Example: 

top
free -m

These commands help us monitor system performance and identify high resource usage.

2) Difference between top, htop, ps, free

top

Shows real-time running processes. Displays CPU usage, memory usage, load average, and running tasks. Updates automatically every few seconds.

htop

Improved version of top. Shows colored output. Easy to scroll and kill processes. More user-friendly.

ps

Shows snapshot of current running processes. It does not update automatically. Example: 

ps aux

free

Shows only memory usage. Displays total, used, free, shared, buffer, and cached memory. Example: 

free -m

3) How to kill a stuck process?

If a process is not responding, we can kill it using the kill command.

Step 1: Find the process ID (PID) 

ps aux | grep process_name

Step 2: Kill the process 

kill PID

If it does not stop, use force kill: 

kill -9 PID

Signal 9 (SIGKILL) forcefully stops the process.

4) What is Load Average?

Load average shows how busy the system is. It tells how many processes are waiting for CPU.

You can check load average using: 

uptime
top

It shows 3 numbers:

  • 1 minute load average
  • 5 minute load average
  • 15 minute load average

If load average is higher than number of CPU cores, system is overloaded. Example: If system has 2 CPU cores and load average is 4, that means system is heavily loaded.

5) What happens when memory is full?

When memory becomes full:

  • System becomes slow.
  • Applications may crash.
  • Swap memory will be used.
  • CPU usage may increase.

If both RAM and Swap are full, Linux may kill processes automatically. This is called OOM (Out Of Memory) Killer.

To check memory usage: 

free -m
top

To fix memory issue:

  • Restart unnecessary services.
  • Kill unused processes.
  • Increase RAM.
  • Optimize application memory usage.

1) Difference Between Hard Link and Soft Link

Hard Link

  • A hard link is another name for the same file.
  • Both original file and hard link share the same inode number.
  • If original file is deleted, hard link still works.
  • Cannot be created for directories (normally).
  • Cannot cross different file systems.

Example:

ln original.txt hardlink.txt

Soft Link (Symbolic Link)

  • A soft link is like a shortcut to another file.
  • It has a different inode number.
  • If original file is deleted, soft link becomes broken.
  • Can link directories.
  • Can cross different file systems.

Example:

ln -s original.txt softlink.txt

In simple words: Hard link = same file, different name. Soft link = shortcut to file.

2) How Linux File Permissions Work

Linux file permissions control who can read, write, or execute a file.

There are 3 types of users:

  • Owner – Person who created the file
  • Group – Users in the same group
  • Others – Everyone else

There are 3 types of permissions:

  • r = Read (4)
  • w = Write (2)
  • x = Execute (1)

Example: 

-rwxr-xr--

Breakdown:

  • Owner: rwx (7)
  • Group: r-x (5)
  • Others: r-- (4)

Numeric permission = 754

To change permission: 

chmod 755 file.txt

To change owner: 

chown user:group file.txt

3) What is umask?

Umask defines the default permission for new files and directories.

When a new file is created:

  • Default file permission = 666
  • Default directory permission = 777

Umask value is subtracted from default permission.

Example: 

umask 022

For file: 666 - 022 = 644

For directory: 777 - 022 = 755

To check current umask: 

umask

So, umask controls security by limiting default permissions.

2) What is Dockerfile?

Dockerfile is a text file that contains instructions to build a Docker image.

It defines:

  • Base image
  • Application files
  • Dependencies
  • Commands to run

Example: 

FROM python:3.9
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
CMD ["python", "app.py"]

Build image: 

docker build -t myapp .

3) Explain Docker Layers

Each instruction in Dockerfile creates a layer.

  • FROM → Base layer
  • RUN → New layer
  • COPY → New layer
  • CMD → Final layer

Layers are cached. If one layer changes, only that layer rebuilds. This makes Docker build faster.

Docker uses layered file system for efficiency.

2) Rebase vs Merge

Merge:

Merge combines two branches and creates a new merge commit. It keeps full history. 

git merge branch_name

Result: Extra merge commit is created.

Rebase:

Rebase moves your branch on top of another branch. It creates cleaner history. 

git rebase branch_name

Key Difference: - Merge keeps original history. - Rebase rewrites history.

Interview Tip: Use merge for shared branches. Use rebase for clean local history.

🚀 AWS EC2 User Data – Complete Guide

🧠 What is User Data?

User Data is a startup script that runs automatically when an EC2 instance starts. It is mainly used to install software, configure services, and prepare the server without manual login.

User Data runs as the root user and is executed during the boot process using a service called cloud-init.

⚙️ How to Use User Data

You add User Data while launching an EC2 instance or by editing it after stopping the instance. The script is written in bash shell format.

Example use cases include installing Docker, Git, Nginx, Java, or starting applications automatically.

🌟 Benefits of User Data

✔ Fully automated server setup

✔ Saves manual installation time

✔ Same configuration for all servers

✔ Useful for auto scaling environments

❓ Can We Modify User Data After Launch?

Yes, User Data can be modified, but it does NOT run automatically again. We must force it to run using cloud-init commands.

🔁 Step-by-Step: Modify and Run User Data

Step 1: Stop the EC2 instance

Step 2: Edit User Data from AWS Console

Step 3: Start the instance

Step 4: Force user data to run again

🔹 Amazon Linux / RHEL Commands

sudo rm -f /var/lib/cloud/instances/*/sem/config_scripts_user
sudo cloud-init clean
sudo reboot

🔹 Ubuntu Commands

sudo cloud-init clean
sudo reboot

✍️ Example User Data Script

#!/bin/bash
yum update -y
yum install docker -y
systemctl start docker
systemctl enable docker

📂 Where to Check User Data Logs

Amazon Linux: /var/log/cloud-init-output.log

Ubuntu: /var/log/cloud-init.log

↓

↓
Start EC2
↓
cloud-init clean
↓
Reboot

EC2 User Data is a startup script that runs during instance boot. We can modify it by stopping the instance, editing the user data, and forcing it to rerun using cloud-init commands.

🐳 Docker Storage Types Explained

Docker containers are temporary by default. When a container is removed, its data is also removed. Docker storage helps us save data outside the container so that it is not lost. Docker provides three main types of storage to handle different use cases.

1️⃣ Docker Volumes

Docker volumes are the most commonly used and recommended storage type in Docker. A volume is created and managed directly by Docker and stored in a special Docker directory. When a container writes data to a volume, the data is stored safely outside the container. Even if the container stops or is deleted, the data inside the volume is not lost. First, a volume is created using a Docker command or automatically during container creation. Next, the volume is attached to a directory inside the container. The container then reads and writes data to that directory normally. Docker volumes work the same on any machine, which makes them reliable. They are easy to back up and restore when needed. Docker volumes are best suited for production applications and databases.

2️⃣ Bind Mounts

Bind mounts directly connect a folder on the host machine to a folder inside the container. In this method, Docker does not manage the storage location. The developer chooses an existing directory on the host system. When the container starts, that directory is mounted inside the container. Any changes made inside the container are immediately reflected on the host. This is very useful during development because code changes appear instantly. Bind mounts make it easy to edit files using local tools. However, they depend heavily on the host file system structure. This makes them less portable between different machines. Bind mounts are mainly used in development environments.

3️⃣ tmpfs Mounts

tmpfs mounts store data directly in the system memory instead of on disk. When a container writes data to a tmpfs mount, it is stored in RAM. This data is very fast to access because it does not involve disk operations. tmpfs storage is temporary and exists only while the container is running. Once the container stops or restarts, all data in tmpfs is removed. This type of storage is useful for sensitive information. It is often used for temporary cache data or secret files. Since data is never written to disk, security is improved. tmpfs mounts reduce disk usage on the host system. They are best used when data does not need to be saved.

🧠 Easy Memory Trick

Docker volumes are managed by Docker and used in production. Bind mounts connect host folders and are mainly used in development. tmpfs mounts store data in memory and are used for temporary data.

🎯 Interview Answer

Docker provides three types of storage: volumes, bind mounts, and tmpfs mounts. Volumes are managed by Docker and used for persistent data. Bind mounts link host directories to containers. tmpfs mounts store temporary data in memory.
