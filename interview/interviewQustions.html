1) How do you check CPU & Memory usage?

In Linux, we can check CPU and memory usage using different commands. The most common commands are:

  • top – Shows real-time CPU and memory usage.
  • htop – Advanced version of top with better UI.
  • free -m – Shows memory usage in MB.
  • ps aux – Shows running processes.
  • vmstat – Shows system performance statistics.

Example: 

top
free -m

These commands help us monitor system performance and identify high resource usage.

2) Difference between top, htop, ps, free

top

Shows real-time running processes. Displays CPU usage, memory usage, load average, and running tasks. Updates automatically every few seconds.

Understanding Linux top Command Output

1. System Uptime & Load Average

top - 16:28:48 up 8 min, 1 user, load average: 0.34, 0.47, 0.31

  • 16:28:48 → Current system time.
  • up 8 min → System has been running for 8 minutes.
  • 1 user → One user is currently logged in.
  • Load Average: 0.34, 0.47, 0.31 → Average system load over the last 1, 5, and 15 minutes.

    These numbers are very low. If your system has 4 or more CPU cores, this means your CPU is under very light load and running smoothly.

2. Tasks (Processes)

Tasks: 347 total, 4 running, 343 sleeping, 0 stopped, 0 zombie

  • 347 total → Total number of processes.
  • 4 running → Currently using CPU.
  • 343 sleeping → Waiting for resources (normal).
  • 0 stopped → No paused processes.
  • 0 zombie → No defunct processes (good sign).

Status: Healthy process state. No zombie processes.

3. CPU Usage

%Cpu(s): 5.5 us, 2.2 sy, 0.0 ni, 92.2 id, 0.1 wa, 0.0 hi, 0.0 si, 0.0 st

  • 5.5% us → CPU used by user processes.
  • 2.2% sy → CPU used by system/kernel processes.
  • 92.2% id → CPU idle (very good).
  • 0.1% wa → Waiting for disk I/O (very low).

Status: CPU is mostly idle. System is not stressed.

4. Memory Usage

MiB Mem: 7776.0 total, 1359.0 free, 3770.0 used, 2964.3 buff/cache

  • 7776 MB total → Total RAM (~8 GB).
  • 1359 MB free → Completely unused RAM.
  • 3770 MB used → Actively used by applications.
  • 2964 MB buff/cache → Used for caching (can be freed if needed).

Linux uses RAM efficiently. Even though free memory looks low, cache memory can be released when applications need it.

5. Swap Usage

MiB Swap: 4096.0 total, 4096.0 free, 0.0 used

  • 4 GB Swap available
  • 0 GB used → Excellent (system is not under memory pressure).

Final System Health Summary

  • ✅ Load Average is low
  • ✅ CPU is 92% idle
  • ✅ No zombie processes
  • ✅ Swap not used
  • ✅ Memory usage is normal

Conclusion

Your system is running smoothly with no performance issues. It is in a healthy state and has plenty of available resources.

Understanding ps aux Command Output

1. What is ps aux?

The ps aux command shows all running processes in the system along with detailed information such as CPU usage, memory usage, process ID, and command.

2. Column Explanation

Column Meaning
USER The owner of the process.
PID Process ID (unique number for each process).
%CPU CPU usage percentage.
%MEM Memory usage percentage.
VSZ Virtual memory size (in KB).
RSS Resident memory (actual RAM used in KB).
TTY Terminal associated with the process.
STAT Process state (Running, Sleeping, etc.).
START Start time of the process.
TIME Total CPU time used by the process.
COMMAND The command that started the process.

3. Understanding Your Output

PID 1 → /sbin/init splash

  • This is the first process started by the Linux kernel.
  • It initializes the system services.
  • Very important system process.

Processes in Square Brackets [ ]

Example: [kthreadd], [kworker/0:1-events], [migration/0]

  • These are kernel threads.
  • They are managed by the Linux kernel.
  • They do background system work (CPU scheduling, memory management, interrupts).
  • They normally show 0% CPU because they run briefly when needed.

4. Common Process States (STAT Column)

  • S → Sleeping (waiting for event) ✅ Normal
  • R → Running
  • I → Idle kernel thread
  • I< → High priority idle kernel thread
  • Ss → Sleeping session leader

5. Why So Many kworker, migration, cpuhp Processes?

You have multiple CPU cores. Each core gets its own kernel threads like:

  • kworker → Handles background kernel tasks
  • migration → Moves tasks between CPU cores
  • ksoftirqd → Handles software interrupts
  • cpuhp → CPU hotplug management

Seeing many of these processes is completely normal on multi-core systems.

6. System Health Based on Your Output

  • Most processes use 0% CPU → System is idle.
  • No abnormal high CPU usage.
  • Kernel threads are running normally.

Final Conclusion

Your system processes look completely normal. The multiple kernel threads are expected in a multi-core Linux system. There is no sign of high CPU usage or abnormal behavior.

4) What is Load Average?

Load average shows how busy the system is. It tells how many processes are waiting for CPU.

You can check load average using: 

uptime
top

It shows 3 numbers:

  • 1 minute load average
  • 5 minute load average
  • 15 minute load average

If load average is higher than number of CPU cores, system is overloaded. Example: If system has 2 CPU cores and load average is 4, that means system is heavily loaded.

Understanding Load Average (CPU Cores Analogy)

Think of your CPU cores as lanes on a bridge. The "Load Average" tells you how many "cars" (processes) are currently on the bridge or waiting in line to get on.

When you see three numbers like 0.09, 0.17, 0.27, they represent the average load over the last 1 minute, 5 minutes, and 15 minutes.

1. The "Bridge" Analogy

To understand if your load is high or low, you first need to know how many CPU cores you have.

If you have 1 CPU core:

  • 0.50: The bridge is half full. Traffic is moving smoothly.
  • 1.00: The bridge is at exactly 100% capacity. It’s perfect efficiency, but any more traffic will cause a delay.
  • 2.00: The bridge is full, and there is a line of cars exactly as long as the bridge itself waiting to get on. Things are slowing down.

If you have 4 CPU cores:

  • 1.00: The bridge is only 25% full. You have 3 empty lanes.
  • 4.00: All lanes are full. You are at 100% capacity.

2. What do the three numbers tell you?

By looking at all three numbers together, you can tell if your system's stress is increasing or decreasing:

Load Trend Meaning
0.10, 0.50, 1.00 Improving: The load was high 15 mins ago, but it is dropping now.
1.00, 0.50, 0.10 Worsening: The system is getting busier. You might have a runaway process.
0.50, 0.50, 0.50 Stable: The workload is consistent and predictable.

3. When should you worry?

A high load average doesn't always mean the CPU is working hard; it can also mean processes are waiting for Disk I/O (waiting for a slow hard drive to read/write data).

  • Safe Zone: Load is below the number of CPU cores (e.g., Load < 4 on a 4-core machine).
  • Caution Zone: Load is equal to the number of cores. You have no "headroom" left for spikes.
  • Danger Zone: Load is significantly higher than the number of cores (e.g., Load of 10.0 on a 2-core machine). Your system will feel laggy or unresponsive.

How to check your "capacity"

Run this command to see how many "lanes" (cores) you have:

grep -c 'processor' /proc/cpuinfo

1) Difference Between Hard Link and Soft Link

Hard Link

  • A hard link is another name for the same file.
  • Both original file and hard link share the same inode number.
  • If original file is deleted, hard link still works.
  • Cannot be created for directories (normally).
  • Cannot cross different file systems.

Example:

ln original.txt hardlink.txt

Soft Link (Symbolic Link)

  • A soft link is like a shortcut to another file.
  • It has a different inode number.
  • If original file is deleted, soft link becomes broken.
  • Can link directories.
  • Can cross different file systems.

Example:

ln -s original.txt softlink.txt

In simple words: Hard link = same file, different name. Soft link = shortcut to file.

2) How Linux File Permissions Work

Linux file permissions control who can read, write, or execute a file.

There are 3 types of users:

  • Owner – Person who created the file
  • Group – Users in the same group
  • Others – Everyone else

There are 3 types of permissions:

  • r = Read (4)
  • w = Write (2)
  • x = Execute (1)

Example: 

-rwxr-xr--

Breakdown:

  • Owner: rwx (7)
  • Group: r-x (5)
  • Others: r-- (4)

Numeric permission = 754

To change permission: 

chmod 755 file.txt

To change owner: 

chown user:group file.txt

3) What is umask?

Umask defines the default permission for new files and directories.

When a new file is created:

  • Default file permission = 666
  • Default directory permission = 777

Umask value is subtracted from default permission.

Example: 

umask 022

For file: 666 - 022 = 644

For directory: 777 - 022 = 755

To check current umask: 

umask

So, umask controls security by limiting default permissions.

2) What is Dockerfile?

Dockerfile is a text file that contains instructions to build a Docker image.

It defines:

  • Base image
  • Application files
  • Dependencies
  • Commands to run

Example: 

FROM python:3.9
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
CMD ["python", "app.py"]

Build image: 

docker build -t myapp .

3) Explain Docker Layers

Each instruction in Dockerfile creates a layer.

  • FROM → Base layer
  • RUN → New layer
  • COPY → New layer
  • CMD → Final layer

Layers are cached. If one layer changes, only that layer rebuilds. This makes Docker build faster.

Docker uses layered file system for efficiency.

🐳 Docker Storage Types Explained

      

        Docker containers are temporary by default. When a container is removed, its data is also removed.
        Docker storage helps us save data outside the container so that it is not lost.
        Docker provides three main types of storage to handle different use cases.
      

    


    
    

      
1️⃣ Docker Volumes

      

        Docker volumes are the most commonly used and recommended storage type in Docker.
        A volume is created and managed directly by Docker and stored in a special Docker directory.
        When a container writes data to a volume, the data is stored safely outside the container.
        Even if the container stops or is deleted, the data inside the volume is not lost.
        First, a volume is created using a Docker command or automatically during container creation.
        Next, the volume is attached to a directory inside the container.
        The container then reads and writes data to that directory normally.
        Docker volumes work the same on any machine, which makes them reliable.
        They are easy to back up and restore when needed.
        Docker volumes are best suited for production applications and databases.
      

    


    
    

      
2️⃣ Bind Mounts

      

        Bind mounts directly connect a folder on the host machine to a folder inside the container.
        In this method, Docker does not manage the storage location.
        The developer chooses an existing directory on the host system.
        When the container starts, that directory is mounted inside the container.
        Any changes made inside the container are immediately reflected on the host.
        This is very useful during development because code changes appear instantly.
        Bind mounts make it easy to edit files using local tools.
        However, they depend heavily on the host file system structure.
        This makes them less portable between different machines.
        Bind mounts are mainly used in development environments.
      

    


    
    

      
3️⃣ tmpfs Mounts

      

        tmpfs mounts store data directly in the system memory instead of on disk.
        When a container writes data to a tmpfs mount, it is stored in RAM.
        This data is very fast to access because it does not involve disk operations.
        tmpfs storage is temporary and exists only while the container is running.
        Once the container stops or restarts, all data in tmpfs is removed.
        This type of storage is useful for sensitive information.
        It is often used for temporary cache data or secret files.
        Since data is never written to disk, security is improved.
        tmpfs mounts reduce disk usage on the host system.
        They are best used when data does not need to be saved.
      

    


    
    

      
🧠 Easy Memory Trick

      

        Docker volumes are managed by Docker and used in production.
        Bind mounts connect host folders and are mainly used in development.
        tmpfs mounts store data in memory and are used for temporary data.

        Bind mounts link host directories to containers.

    It is responsible for starting, stopping, managing services, handling boot process, logs, and system states.
  


  
👉 It is the first process (PID 1) started by the Linux kernel.


  
🔹 Why systemd is used?


  
    
    
  • Faster boot (parallel service startup)
    • 
    
  • Better service dependency management
    • 
    
  • Centralized logging
    • 
    
  • Easy service control
    •

      targets

  • Easy environment & volume management
    • 
    
  • Consistent setup across environments
    •

9. Jenkins Stages




  
🔹 What are Jenkins Stages?


  

    In Jenkins Pipeline, stages represent different steps of the CI/CD process such as build, test, and deploy.
    Each stage groups related tasks and helps visualize the pipeline flow.
  


  
🔹 Common Jenkins Pipeline Stages


  
1️⃣ Checkout / Source Code


  
Pull code from Git repository


  
stage('Checkout') {
    steps {
        git 'https://github.com/org/repo.git'
    }
}
  


  
2️⃣ Build


  
Compile or build application


  
stage('Build') {
    steps {
        sh 'mvn clean package'
    }
}
  


  
3️⃣ Test


  
Run unit or integration tests


  
stage('Test') {
    steps {
        sh 'mvn test'
    }
}
  


  
4️⃣ Code Quality / Scan


  
SonarQube, SAST, lint checks


  
stage('Code Quality') {
    steps {
        sh 'sonar-scanner'
    }
}
  


  
5️⃣ Package / Artifact


  
Create WAR/JAR/Docker image


  
stage('Package') {
    steps {
        sh 'docker build -t myapp:latest .'
    }
}
  


  
6️⃣ Push Artifact


  
Push to Nexus / Docker Hub / ECR


  
stage('Push Image') {
    steps {
        sh 'docker push myapp:latest'
    }
}
  


  
7️⃣ Deploy


  
Deploy to server / Kubernetes / ECS


  
stage('Deploy') {
    steps {
        sh 'kubectl apply -f deployment.yaml'
    }
}
  


  
🔹 Typical Jenkins Pipeline Flow


  
Checkout → Build → Test → Scan → Package → Push → Deploy


  
🔹 Why Stages are Important?


  
    
    
  • Clear pipeline visualization
    •

13. Auto Scaling and Load Balancer




  

    
🔹 How to Use Auto Scaling with Load Balancer (AWS)

  


  

    
🔹 Concept First (Simple)

    
Load Balancer (ALB/ELB) distributes incoming traffic.

    
Auto Scaling Group (ASG) automatically adds or removes EC2 instances.

    
CloudWatch triggers scaling actions based on metrics.

    
👉 Together, they provide high availability and scalability.

  


  

    
🔹 Architecture Flow

    
User Traffic

    
↓

    
Load Balancer

    
↓

    
Auto Scaling Group

    
↓

    
EC2 Instances

  


  

    
🔹 Step-by-Step Setup

  


  

    
1️⃣ Create Launch Template / Launch Configuration

    
Defines:

    
    
      
  • AMI
    • 
      
  • Instance type
    • 
      
  • Security Group
    • 
      
  • Key pair
    • 
      
  • User data (application setup)
    • 
    

    
Example User Data:

    
#!/bin/bash
yum install nginx -y
systemctl start nginx
    

  


  

    
2️⃣ Create Target Group

    
Used by the Load Balancer to route traffic.

    
    
      
  • Automatically registers EC2 instances
    • 
      
  • Protocol: HTTP
    • 
      
  • Port: 80
    • 
      
  • Health check path: /
    • 
    

  


  

    
3️⃣ Create Load Balancer (ALB Recommended)

    
    
      
  • Attach Target Group
    • 
      
  • Enable health checks
    • 
      
  • Deploy across multiple Availability Zones
    • 
    

    
👉 ALB routes traffic only to healthy instances.

  


  

    
4️⃣ Create Auto Scaling Group

    
Attach:

    
    
      
  • Launch Template
    • 
      
  • Load Balancer Target Group
    • 
      
  • VPC and Subnets (Multi-AZ)
    • 
    

    
Capacity Settings:

    
    
      
  • Desired: 2
    • 
      
  • Minimum: 1
    • 
      
  • Maximum: 5
    • 
    

  


  

    
5️⃣ Configure Scaling Policies


    
🔸 Scale Out (Increase Instances)

    
    
      
  • Metric: CPU Utilization
    • 
      
  • Condition: CPU > 70%
    • 
      
  • Action: Add 1 EC2 instance
    • 
    


    
🔸 Scale In (Decrease Instances)

    
    
      
  • Metric: CPU Utilization
    • 
      
  • Condition: CPU < 30%
    • 
      
  • Action: Remove 1 EC2 instance
    • 
    


    
CloudWatch automatically triggers scaling actions.

  


  

    
🔹 How It Works in Real Time

    
    
      
  • Traffic increases
    • 
      
  • Load Balancer distributes traffic
    • 
      
  • CPU usage increases
    • 
      
  • CloudWatch alarm triggers
    • 
      
  • Auto Scaling launches new EC2
    • 
      
  • EC2 auto-registers with Load Balancer
    • 
      
  • Traffic becomes balanced 🎯
    • 
    

  


  

    
🔹 Health Check Role

    

      

        Component
        Purpose
      

      

        Load Balancer
        Checks application health
      

      

        Auto Scaling
        Replaces unhealthy instances
      

    

    
👉 If an instance fails, ASG terminates it and launches a new one automatically.

  


  

    
🔹 Benefits

    
    
      
  • High availability
    • 
      
  • Automatic scaling
    • 
      
  • Fault tolerance
    • 
      
  • No manual intervention
    • 
      
  • Cost optimization
    • 
    

  


  

    
🔹 Interview-Ready Answer (Short)

    

      We use Auto Scaling with a Load Balancer to automatically scale EC2 instances
      based on traffic and CloudWatch metrics. The Load Balancer distributes traffic,

15. What is IAM




  

    
🔹 What is IAM (Identity and Access Management)?

    

      IAM is an AWS service used to manage users, permissions, and access to AWS resources securely.
    

    
👉 In simple words: IAM controls who can access what in AWS.

  


  

    
🔹 Why do we use IAM?

    
    
      
  • Secure AWS account access
    • 
      
  • Avoid using root account
    • 
      
  • Give least privilege access
    • 
      
  • Control permissions for users & services
    • 
    

  


  

    
🔹 IAM Components


    
1️⃣ Users

    
Individual people

    
Example: Developer, Admin


    
2️⃣ Groups

    
Collection of users

    
Example: DevOps team


    
3️⃣ Roles (Very Important)

    
Used by AWS services

    
Temporary permissions

    
Example:

    
    
      
  • EC2 role to access S3
    • 
      
  • Lambda role to access DynamoDB
    • 
    


    
4️⃣ Policies

    
JSON documents defining permissions

    
Example:

    
{
  "Effect": "Allow",
  "Action": "s3:ListBucket",
  "Resource": "*"
}
    

  


  

    
🔹 Real-Time Example

    
👉 EC2 needs access to S3

    
    
      
  • ❌ Don’t use access keys
    • 
      
  • ✅ Attach IAM Role to EC2
    • 
    

  


  

    
🔹 IAM Best Practices

    
    
      
  • Never use root account for daily work
    • 
      
  • Enable MFA
    • 
      
  • Use roles instead of access keys
    • 
      
  • Follow least privilege principle
    • 
      
  • Rotate credentials regularly
    • 
    

  


  

    
🔹 Interview-Ready Answer (Short)

    

      IAM is an AWS service used to manage access to AWS resources by defining users, roles, and permissions securely.

      affecting system performance, availability, or functionality.

    •

    •

    •

    •

    • 
      
  • Restart Services or Server if Needed: systemctl restart <service>, reboot
    •

    • 
      
  • Checked Docker container logs: docker logs webapp
    •

    •

    • 
      
  • Cleared old logs: rm -rf /var/log/old_logs/*
    • 
      
  • Set CloudWatch alarms to monitor CPU and disk usage
    •

🔹 Shell Scripting vs Bash Scripting

    

      

        Feature
        Shell Scripting
        Bash Scripting
      

      

        Definition
        Writing scripts for any Unix/Linux shell (sh, csh, ksh, zsh) to automate tasks
        Writing scripts specifically for the Bash shell (Bourne Again Shell)
      

      

        Shell Type
        Can be for any shell (sh, csh, ksh, zsh)
        Only for Bash shell
      

      

        Portability
        Scripts may run on different shells if compatible
        Scripts are specific to Bash features
      

      

        Features
        Basic scripting features: loops, conditions, variables
        Advanced features: arrays, associative arrays, string manipulation, arithmetic operations
      

      

        Default on Linux
        Depends on the system (sh or dash)
        Usually installed by default as /bin/bash on most Linux systems
      

      

        Syntax Differences
        Limited syntax support; may differ per shell
        Bash syntax includes more enhancements and is more flexible
      

      

        Example
        #!/bin/sh
        #!/bin/bash
      

    

  


  

    
🔹 Simple Explanation

    
Shell scripting → Writing scripts for any shell to automate tasks.

    
Bash scripting → Writing scripts using Bash-specific features, more powerful and widely used.

  


  

    
🔹 Real-Time Example

    
Shell Script (sh):

    
#!/bin/sh
echo "Hello from shell script"
    

    
Bash Script (bash):

    
#!/bin/bash
echo "Hello from Bash script"
array=(one two three)
for i in "${array[@]}"; do
  echo $i
done
    

  


  

    
🔹 Interview-Ready Short Answer

    

      Shell scripting is writing scripts for any Unix/Linux shell to automate tasks, 
      while Bash scripting is specifically for the Bash shell, offering more advanced features, 
      better syntax, and wide usage in Linux automation.

🔹 Types of Load Balancers

    

      

        Type
        Description
        Layer
        Use Case
        Example
      

      

        Application Load Balancer (ALB)
        Routes traffic based on content (URL, headers, path)
        Layer 7 (HTTP/HTTPS)
        Microservices, HTTP apps, API routing
        AWS ALB, NGINX
      

      

        Network Load Balancer (NLB)
        Routes traffic based on IP protocol data
        Layer 4 (TCP/UDP)
        High-performance, low-latency, TCP/UDP apps
        AWS NLB, HAProxy
      

      

        Classic Load Balancer (CLB)
        Basic load balancer, supports HTTP, HTTPS, TCP
        Layer 4 & Layer 7
        Legacy applications
        AWS ELB (Classic)
      

      

        Global Load Balancer
        Distributes traffic across multiple regions
        Layer 7
        Multi-region apps, disaster recovery
        AWS Global Accelerator, Cloudflare
      

      

        Internal / Private Load Balancer
        Load balances traffic inside VPC/private network
        Layer 4 or 7
        Internal microservices
        AWS ALB internal, NGINX
      

      

        Hardware Load Balancer
        Physical device for traffic distribution
        Layer 4 & 7
        Data centers, enterprise apps
        F5 Big-IP, Citrix ADC
      

      

        Software Load Balancer
        Software-based, can run on VMs or containers
        Layer 4 & 7
        Cloud-native apps
        HAProxy, NGINX, Traefik
      

    

  


  

    
🔹 Simple Explanation

    
Load balancers distribute incoming network traffic across multiple servers to ensure high availability and reliability.

    
ALB → Smart, content-based routing (HTTP/HTTPS).

    
NLB → Fast, TCP/UDP-based, low-latency routing.

    
CLB → Old-style basic balancing.

  


  

    
🔹 Real-Time Example

    
    
      
  • Your web application has 3 EC2 instances running.
    • 
      
  • User requests come in; ALB distributes requests evenly based on HTTP path.
    • 
      
  • If one instance fails, ALB stops sending traffic to it.
    • 
      
  • NLB can be used for TCP-based apps like game servers or database connections.
    • 
    

  


  

    
🔹 Interview-Ready Short Answer

    

      Load balancers distribute incoming traffic across multiple servers to ensure high availability, scalability, and fault tolerance. 
      Types include Application Load Balancer (Layer 7), Network Load Balancer (Layer 4), Classic Load Balancer, 
      Global Load Balancer, Internal Load Balancer, and hardware/software load balancers.
