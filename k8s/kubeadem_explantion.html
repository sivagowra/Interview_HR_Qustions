Kubernetes Cluster Setup Using kubeadm – Clear Explanation

What This Script Does (Simple View)

This script prepares a Linux server to become:

  • Control Plane (Master Node)
  • OR Worker Node

It is called Common Setup because it runs on both nodes.

It prepares the system by:

  • Cleaning and updating system
  • Fixing memory settings
  • Fixing networking settings
  • Installing container runtime
  • Installing Kubernetes tools

STEP 1: Update System

sudo yum update -y

Updates all installed packages to latest versions.

  • Fixes security issues
  • Removes bugs
  • Makes system stable

Before building Kubernetes cluster, system must be clean and updated.

STEP 2: Disable Swap

sudo swapoff -a sudo sed -i '/ swap / s/^\(.*\)$/#\1/g' /etc/fstab

What is Swap?
Swap is extra memory stored on disk.

Why disable swap?

  • Kubernetes needs exact memory control
  • If swap is enabled, kubeadm will fail
  • Pods may behave unpredictably

First command disables swap immediately.
Second command disables swap permanently.

STEP 3: Disable SELinux

sudo setenforce 0 sudo sed -i 's/^SELINUX=enforcing/SELINUX=permissive/' /etc/selinux/config

SELinux is a Linux security module.

Strict SELinux rules may block:

  • Container networking
  • Kubernetes communication

So we set it to permissive mode.

STEP 4: Load Kernel Module (br_netfilter)

sudo modprobe br_netfilter

This allows Linux to see container network traffic.

Without this:

  • Pods cannot communicate
  • Services may not work

STEP 5: Configure Network Settings

net.bridge.bridge-nf-call-iptables=1
net.bridge.bridge-nf-call-ip6tables=1
net.ipv4.ip_forward=1

This enables:

  • IP forwarding
  • Bridge network filtering

Pods use IP addresses to communicate.
Without IP forwarding, pod networking fails.

STEP 6: Install containerd

sudo yum install -y containerd

containerd is the container runtime.

Kubernetes does not run containers directly.
It uses container runtime like containerd.

STEP 7: Configure containerd

SystemdCgroup = true

Kubelet uses systemd to manage resources.

If containerd uses different cgroup driver:

  • Cluster may fail
  • Node may become NotReady

So both must match.

STEP 8: Install Kubernetes Components

sudo yum install -y kubelet kubeadm kubectl

Each component:

  • kubeadm → Initializes cluster
  • kubelet → Runs pods on each node
  • kubectl → Command line tool to control cluster

Final Understanding

After this script runs:

  • System is ready for Kubernetes
  • Cluster is NOT created yet

Next steps:

  • On Master Node → Run kubeadm init
  • On Worker Node → Run kubeadm join <token>

Simple Example

  • kubeadm → Setup tool (Principal)
  • kubelet → Node manager (Teacher)
  • containerd → Runs containers (Classroom)
  • kubectl → Control panel (Office)

In One Line

This script prepares a Linux machine to safely run a Kubernetes cluster.

Kubernetes Master Node Setup (kubeadm init + Networking)

STEP 1: Initialize Kubernetes Cluster

sudo kubeadm init --pod-network-cidr=10.244.0.0/16

What this does: This command initializes the Kubernetes Control Plane (Master Node).

It performs the following internally:

  • Creates Kubernetes control plane components
  • Starts API Server
  • Starts Scheduler
  • Starts Controller Manager
  • Generates cluster certificates
  • Creates admin configuration file
  • Generates worker join token

Why --pod-network-cidr=10.244.0.0/16?

  • This defines the IP range for Pods
  • Required for Flannel network plugin
  • Without this, networking will fail

After successful execution:

  • Cluster control plane becomes ready
  • You receive kubeadm join command for worker nodes

STEP 2: Install Flannel (Pod Network)

kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml

What is Flannel? Flannel is a CNI (Container Network Interface) plugin.

Kubernetes does NOT provide networking by default. You must install a network plugin.

Flannel provides:

  • Pod-to-Pod communication
  • Overlay network between nodes
  • IP allocation for Pods

Why required?

  • Without Flannel, Pods cannot communicate
  • Nodes will show NotReady status

After installation:

  • Node status becomes Ready
  • Pod networking starts working

STEP 3: Install MetalLB (LoadBalancer Support)

kubectl apply -f https://raw.githubusercontent.com/metallb/metallb/v0.13.12/config/manifests/metallb-native.yaml

What is MetalLB? MetalLB provides LoadBalancer service support for bare-metal Kubernetes clusters.

In cloud environments (AWS, Azure, GCP): LoadBalancer is provided automatically.

But in local or on-premise setup: LoadBalancer does not work by default.

MetalLB solves this by:

  • Assigning external IP addresses
  • Managing load balancer IP pools
  • Allowing Service type LoadBalancer to work

After installing MetalLB:

  • You must configure IP address pool
  • Then LoadBalancer services will get external IPs

Complete Flow Understanding

  • kubeadm init → Creates Control Plane
  • Flannel → Enables Pod networking
  • MetalLB → Enables LoadBalancer support

Now your cluster is fully functional.

What Happens After These Steps?

  • Run kubeadm join on worker nodes
  • Deploy applications
  • Create services
  • Expose services using LoadBalancer

In Simple Words

  • kubeadm init → Builds the brain (Master)
  • Flannel → Connects all Pods
  • MetalLB → Gives external access

Now your Kubernetes cluster is fully operational.
