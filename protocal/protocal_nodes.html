Most Important Core Protocols and Ports in Networking

Core Protocols and Their Ports

1. FTP (File Transfer Protocol)

  • Purpose: Transfer files between client and server.
  • Port: TCP 21 (control), TCP 20 (data).
  • How it works: Client connects to FTP server, authenticates, then uploads/downloads files.
  • Example: A web developer uploads website files to a hosting server.

2. SSH (Secure Shell)

  • Purpose: Secure remote login and command execution.
  • Port: TCP 22.
  • How it works: Encrypts all traffic (unlike Telnet). Used mainly in Linux/Unix.
  • Example: A system admin logs into a Linux server from home securely.

3. Telnet

  • Purpose: Remote login and command execution (insecure).
  • Port: TCP 23.
  • How it works: Sends data in plain text, easily intercepted. Rarely used today.
  • Example: In the past, admins managed routers/switches with Telnet (now replaced by SSH).

4. SMTP (Simple Mail Transfer Protocol)

  • Purpose: Sending email between mail servers.
  • Port: TCP 25 (standard), TCP 587 (submission), TCP 465 (secure, legacy).
  • How it works: When you send an email, your client uses SMTP to deliver it to the mail server.
  • Example: Gmail sending an email to Outlook.com.

5. DNS (Domain Name System)

  • Purpose: Translates domain names to IP addresses.
  • Port: UDP 53 (queries), TCP 53 (zone transfers).
  • How it works: You type google.com, DNS resolves it to an IP (e.g., 142.250.72.206).
  • Example: Without DNS, you'd have to remember IPs for every website.

6. DHCP (Dynamic Host Configuration Protocol)

  • Purpose: Automatically assigns IP addresses and network configs.
  • Port: UDP 67 (server), UDP 68 (client).
  • How it works: Client sends a broadcast "I need an IP," DHCP server replies with IP, subnet mask, gateway, DNS.
  • Example: Your phone automatically gets an IP when joining Wi-Fi.

7. HTTP (HyperText Transfer Protocol)

  • Purpose: Web browsing (not secure).
  • Port: TCP 80.
  • How it works: Client (browser) requests a webpage, server responds with HTML.
  • Example: Visiting a website using `http://`.

8. HTTPS (HyperText Transfer Protocol Secure)

  • Purpose: Secure web browsing (encrypted with SSL/TLS).
  • Port: TCP 443.
  • How it works: Same as HTTP but with encryption, which prevents eavesdropping.
  • Example: Online banking, shopping, secure logins (`https://`).

9. POP3 (Post Office Protocol v3)

  • Purpose: Retrieve emails from mail server (downloads and usually deletes from server).
  • Port: TCP 110 (POP3), TCP 995 (POP3S secure).
  • How it works: Downloads emails to local device, not stored on server.
  • Example: Old email clients like Outlook Express.

10. IMAP (Internet Message Access Protocol)

  • Purpose: Retrieve emails, but keeps them stored on the server.
  • Port: TCP 143 (IMAP), TCP 993 (IMAPS secure).
  • How it works: Synchronizes emails across devices (server keeps master copy).
  • Example: Gmail app syncing email across your phone, tablet, and laptop.

Summary of Protocol Types

  • FTP, HTTP, SMTP = transfer protocols (files, web pages, emails).
  • SSH, Telnet = remote access protocols.
  • DNS, DHCP = network services.
  • POP3, IMAP = email retrieval protocols.
  • HTTPS = secure version of HTTP.

NetBIOS (Network Basic Input/Output System)

  • It's an API that allows applications on different computers to communicate over a local network (created in the 1980s).
  • It is a set of functions (like name resolution, session management, data transfer), not a protocol by itself.

NetBIOS over TCP/IP (NetBT)

  • NetBT is the method of running NetBIOS services on top of the TCP/IP stack.
  • This allows NetBIOS-based applications (like older Windows file sharing) to work across larger routed networks.

Ports used by NetBIOS/NetBT:

  • UDP 137 → NetBIOS Name Service (name registration and resolution)
  • UDP 138 → NetBIOS Datagram Service (connectionless communication, broadcasts)
  • TCP 139 → NetBIOS Session Service (connection-oriented communication, legacy file/printer sharing)

Lightweight Directory Access Protocol (LDAP)

  • LDAP is an open, standard protocol used to access and manage directory services (like a phonebook for network resources).
  • In plain words: It is the language applications use to look up info about users, computers, groups, etc., on a directory server.

Ports used by LDAP

  • TCP/UDP 389 → Standard LDAP
  • TCP 636 → LDAPS (LDAP over SSL/TLS, secure)

LDAP vs Active Directory (AD)

  • Active Directory (AD) is Microsoft's directory service (the database/service itself).
  • LDAP is the protocol (the language) used to talk to directory services like AD for authentication, authorization, and lookups.

Ports in AD (using LDAP)

  • 389 (LDAP) - Clear text queries/authentication.
  • 636 (LDAPS) - Secure LDAP over SSL/TLS.
  • 3268 - Global Catalog LDAP (for searching across multiple domains).

Server Message Block (SMB) and CIFS

  • SMB is the network file sharing protocol that makes "Windows File Sharing" work (i.e., accessing `\\Server\SharedFolder`).
  • CIFS (Common Internet File System) is Microsoft's specific and now obsolete (legacy) implementation of SMB version 1.0.
  • Modern systems use newer versions (SMB 3.x) which add encryption and performance improvements.

Ports Used by SMB

  • TCP 445 → Direct SMB over TCP (modern use).
  • TCP 139 → SMB over NetBIOS Session Service (legacy use).

Remote Desktop Protocol (RDP)

  • RDP is a Microsoft protocol that allows a user to connect to another computer over the network and control its full desktop environment remotely.
  • It is essentially secure screen sharing + input redirection.

Ports Used by RDP

  • TCP 3389 → Default RDP port.
  • UDP 3389 → Used in newer versions for performance.

Quick Comparison (RDP vs SMB/SSH/VNC)

Transport Layer Protocols: TCP and UDP

Feature TCP (Transmission Control Protocol) UDP (User Datagram Protocol)
Type Connection-oriented (requires handshake) Connectionless (no handshake)
Reliability Reliable (Ensures delivery, order, and checks for errors) Unreliable (No guarantee of delivery or order)
Speed Slower (due to overhead for reliability) Faster (minimal overhead)
Use Case When accuracy matters (web, file transfer, emails) When speed matters (streaming, calls, games)
Analogy Registered letter with tracking Regular postcard

Application Layer Protocols (Layer 7)

These protocols (HTTP, SMTP, FTP, DNS, etc.) sit above TCP/UDP and define the rules for how applications interpret the data. TCP/UDP provide *delivery*, while Application Protocols provide *context and meaning*.
